# Spec Changes

Semantic changes to the specification. Git history captures diffs; this file captures intent.

## 2026-06-08

- **Entrypoint migration removes the deprecated `@gaodes/pi-graphify` from existing Pi settings on boot** ([REQ-AGENT-023](agents.md#req-agent-023-knowledge-graph-capability-graphify)). The graphify rebuild replaced the third-party wrapper with the first-party `graphify-native.ts` and dropped `@gaodes` from `npm/package.json`, the entrypoint install list, and `manifest.json` — but the entrypoint's Pi settings-merge PRESERVES existing `settings.json` packages, so a deployment upgrading from the `@gaodes` era kept `npm:@gaodes/pi-graphify` in its settings and then loaded BOTH it and `graphify-native.ts` → `Tool "graphify_query" conflicts` → Pi refused to start (hit live this session on a fresh `pi` launch). Added a `removedPackageIds` set to the settings-merge (`entrypoint.sh`) that hard-drops `@gaodes/pi-graphify` from existing settings on every boot — dropped and never re-added, distinct from the disabled-package path that re-adds context-mode as a disabled spec — so upgrades self-heal. Settings removal alone is sufficient (Pi only loads packages listed in settings); the stale `node_modules/@gaodes` files are left untouched (harmless, and the dir may be a preseed symlink). **Status:** REQ-AGENT-023 stays Implemented.
- **Roll-forward and stale-discard now emit a `review_superseded` audit event** ([REQ-AGENT-055](agents.md#req-agent-055-pi-pr-boundary-review-window-advancement), [REQ-AGENT-058](agents.md#req-agent-058-pr-boundary-review-reconciliation-and-missed-event-recovery)). When a new push advanced the PR head while a prior head's review was still in flight, `rollForwardAdvancedReview` abandoned the old window (and killed its lanes) for a fresh review at the new head; the durable event log (`.git/codeflare-review-events.jsonl`) then showed `lane_spawned <oldhead>` followed by nothing, so a deliberate supersede was indistinguishable from a crash (observed live on the `7de81d3`→`4f29529` roll-forward). Both supersede paths (`rollForwardAdvancedReview` and `discardStale`) now `appendReviewEvent({event:"review_superseded", head:<oldHead>, reason, lanes})`, so the audit trail explains every abandoned window — preserving the "a surprising review is one `cat events.jsonl` away" guarantee. Diagnostic-only; no behavior change beyond the emitted event. **Status:** REQ-AGENT-055/058 unchanged (REQ-058 stays Partial).
- **Pi durable review now finalizes fully on an idle session — the autonomous timer emits the merged summary into the session and starts the autofix, not just a disk ack** ([REQ-AGENT-061](agents.md#req-agent-061-pi-idle-durable-review-reaper) AC3). The detached-lane reaper timer only ack'd + persisted a `summary.md` to disk when an idle session's lanes all finished — it could not emit the merged summary into the Pi conversation or start the autofix, because both need a live `pi`/`ctx` and the module-level `setInterval` has neither. So an idle completed review still required the user to submit a prompt before any results appeared or fixing began (two user-reported defects: results never shown, Pi never resumed). Fix: the default export binds a module-level `activeReviewFinalize` hook to a `pi`-aware finalize closure; on all-lanes-complete the timer calls it → `finalizeCompletedReview` (made `ctx`-optional) runs the existing claim-guarded `publishReviewSummary` (`pi.sendMessage({display:true})`) and `requestReviewAutofix` (`pi.sendMessage({triggerTurn:true})`), so the summary is emitted and the autofix turn starts with zero user input (the user cancels with ESC). The `ctx`-only paths degrade safely: the summary-emit fallback notify is skipped, the autofix manual-mode suppression defaults off (ESC-to-cancel replaces it), and the statusline clear is already try/caught. Dedup markers (`claimReviewAnnouncement` `summary-v3` + the `autofix.requested` file) mean a later real turn never double-emits or double-fires. Pi-agent-only; no Worker runtime path changes. **Status:** REQ-AGENT-061 stays Implemented (the idle-finalization wiring is an integration path verified live in Pi; the decision/parse/recovery core stays unit-tested, with the documented `setInterval`/OS-spawn coverage-gaps).

- **Three review/graphify HIGH findings from the in-session review fixed** ([REQ-AGENT-058](agents.md#req-agent-058-pr-boundary-review-reconciliation-and-missed-event-recovery) AC3 corrected; [REQ-AGENT-023](agents.md#req-agent-023-knowledge-graph-capability-graphify)). The live PR #505 reviewers (running through the new state machine) surfaced three HIGHs against the review/graphify extensions. (1) **`resolveEnforcedHead` armed reviews for unpushed local commits** — it preferred local HEAD whenever the current branch matched the PR branch and local descended from GitHub's reported head, which also matches ordinary unpushed WIP, so passive reconciliation could start a PR-boundary review for a commit the PR never had. (2) **graphify `resolveGraph` could query the wrong graph** — it consulted only the advanced-mode-only active-repo sentinel then the global graph, so in default-mode sessions and review lanes (sentinel absent or stale) `graphify_query` ignored the current repo's `graphify-out/graph.json`. (3) **a text-matching test masqueraded as behavioral** — `agent-seed-manifest.test.ts`'s missed-boundary-recovery test only asserted the generated seed *contained* `reconcileOpenPrReview`/`shouldReconcileOpenPr`, so it would pass even if reconciliation were dead code; converted to exercise the real `shouldReconcileOpenPr` decision (reconciles a real open/enforced/unacked PR; never from passive branch existence or an already-handled head), keeping one honest regression guard that the old unconditional catch-up string stays absent. Pi-agent-only; no Worker runtime path changes. **Status:** REQ-AGENT-058 stays Partial; REQ-AGENT-023 stays Implemented.
- **Pi durable review now self-finalizes on an idle session — autonomous reaper timer added** ([REQ-AGENT-061](agents.md#req-agent-061-pi-idle-durable-review-reaper) AC3 added; [REQ-AGENT-054](agents.md#req-agent-054-pi-durable-review-lane-failure-handling) AC1/AC4/AC6 restructured for the per-AC word cap; [REQ-AGENT-059](agents.md#req-agent-059-pi-durable-review-fix-loop) AC1/AC2 trimmed). Live-testing the detached-lane rewrite (entry below) surfaced a gap: the disk reaper only ran on Pi lifecycle ticks (`agent_end` / turn boundaries / `resources_discover`), and Pi exposes no periodic/idle hook — so a push followed by an idle session left both lanes finished on disk (`agent_end` + findings in the transcript) but unharvested for ~17 minutes (no result file, job stuck `running`, no merged summary) until the user happened to interact. Fix: `review-enforcement.ts` registers one reload-safe, `unref`'d `setInterval` (~20s, cleared-then-reset via a `globalThis` singleton so `/reload` never stacks timers) running `autonomousReviewReaperTick` → it reaps finished lanes from disk and spawns the next *fresh* eligible lane (e.g. doc-updater once spec-reviewer has a result) with no ctx; failed-lane RETRIES stay on the on-turn driver so the circuit breaker still bounds them, and the ctx-dependent merged chat summary + ack still post on the next real interaction. Best-effort (never throws) and a cheap no-op when no review window is pending. Same-turn spec-conciseness pass to satisfy the 45-word per-AC cap (the rewrite ACs below ran long): REQ-054 AC1 absorbs the over-budget process-group kill, AC4 absorbs the `/proc` start-time pid-identity check, AC6 narrows to the completion-write decision (`agent_end`, or a usable final message flushed before exit, → result file), and REQ-AGENT-061 captures the lifecycle and idle driver; REQ-AGENT-059 AC1/AC2 are trimmed to the bare assertion (the stripped rationale is already recorded in the entry below, and AC6's dropped "next review uses the fix diff" clause is already covered by [REQ-AGENT-055](agents.md#req-agent-055-pi-pr-boundary-review-window-advancement) AC4). The interval driver is runtime-only (a `setInterval`), so it carries a documented coverage-gap alongside the existing OS-spawn smoke-test gap. **Status:** REQ-AGENT-061 stays Implemented (timer + OS-spawn layer smoke-tested with documented coverage-gaps); REQ-AGENT-059 unchanged (Implemented).
- **Pi durable review lanes rebuilt from in-process agent sessions to detached child `pi` processes reaped from disk — lanes now actually run to completion and survive the spawning session** ([REQ-AGENT-060](agents.md#req-agent-060-pi-durable-review-lane-tool-surface) AC1 rewritten, [REQ-AGENT-054](agents.md#req-agent-054-pi-durable-review-lane-failure-handling) AC1/AC4 rewritten + AC6 added). Lanes previously ran as in-process `createAgentSession` calls inside the spawning Pi session (`runDurableLane`): when that session exited — the user quitting, or a single-shot run ending — the in-process async work died mid-stream and its in-process `setTimeout` watchdog died with it, so the lane froze at `running` forever with a partial transcript, no `lane_completed`/`lane_failed`, and the watchdog never fired (confirmed from a real frozen on-disk job: the code-reviewer transcript ended on a `thinking_delta` with no pi process alive). The orphan was never reaped, so `recoverDurableReviewLaneState` reset its `running`→`pending` and reconciliation re-spawned it every session, churning to a `breaker_opened`. Pi's own canonical subagent pattern (`examples/extensions/subagent`: "each subagent runs in a separate pi process") is the fix: `spawnDurableLane` now launches each lane as a DETACHED, headless child `pi --mode json -p --no-session --no-extensions --no-context-files --tools read,grep,find,bash --append-system-prompt <agent-prompt>` (stdin `/dev/null`), `unref`'d so it outlives the spawning session and writes its event stream to the lane transcript. A new disk-driven reaper (`reapDurableReviewLanes` + the pure `reapLaneDecision`/`summarizeLaneTranscript`) — run by the `agent_end` driver and `refreshReviewStatusFromDurable` on every tick — is the single `running`→`completed`/`failed` authority: `agent_end` + usable final message → write the result file, mark completed, kill the child group; child dead without `agent_end` or over the 15-min budget → mark failed (over-budget killed). Liveness is the child pid persisted on disk and **identity-checked against the child's `/proc/<pid>/stat` start-time** (Linux recycles pids), so a recycled pid reads as dead rather than a live impostor — preventing both a wedged `running` lane and a SIGKILL to an unrelated process group; a group is killed ONLY when verified-alive-and-ours, and only on the over-budget branch (a completed/dead child is already gone). A usable final assistant message is preserved even if the child dies before flushing the terminal `agent_end` line (no lost reviews). Any session reaps a lane another (now-exited) session spawned — the in-memory `runningLanes` Set is deleted, and `recoverDurableReviewLaneState` no longer resets `running`→`pending` (the churn source). (PID-identity, lost-result, and needless-kill hardening came from the in-session code review of this change.) **Trade-off (supersedes the same-day "durable review lanes load graphify-native.ts directly" note below): lanes run with `--no-extensions`, so a reviewer no longer has graphify/context-mode/`codeflare-pi` inside the lane** — it reviews with `read`/`grep`/`find`/`bash` and the prompt's explicit "do not run builds/tests/linters" instruction. `--no-extensions` is required for correctness (no recursive `review-enforcement` load) and is what lets the lane child start fast and exit cleanly (verified: a bare in-extension `pi -p` lingered until killed; `--no-extensions` exits 0). Two env requirements proven against pi 0.78.1 and encoded: stdin MUST be `/dev/null` (an inherited stdin hangs print mode at startup with zero output), and completion is detected from `agent_end` in the transcript, never from process exit. New behavioural tests in `agent-seed-manifest.test.ts` (`reapLaneDecision` transition table, `summarizeLaneTranscript` parsing, `recoverDurableReviewLaneState` reflective recovery); the spawn + pid-liveness + group-kill OS layer is an integration smoke test (documented coverage-gaps on REQ-060 AC1 / REQ-054 AC6), since the Workers vitest pool cannot spawn pi. **Status:** REQ-AGENT-060 and REQ-AGENT-054 stay Implemented (decision/parse/recovery logic unit-tested; OS-spawn runtime behavior smoke-tested with documented coverage-gaps, matching the existing runtime-coverage convention).
- **Pi graphify rebuilt as a first-party native extension; the `@gaodes/pi-graphify` third-party wrapper and the dead Pi `mcp.json` are removed** ([REQ-AGENT-023](agents.md#req-agent-023-knowledge-graph-capability-graphify), [REQ-AGENT-024](agents.md#req-agent-024-advanced-session-mode-graph-first-discipline)). Pi has no MCP client, so graphify is now registered by a self-contained `preseed/agents/pi/extensions/graphify-native.ts` (`graphify_query`/`graphify_path`/`graphify_explain` via `pi.registerTool`, mirroring `browser-run.ts`) that shells the SAME upstream `graphify` CLI Claude's MCP server runs (`graphify.serve._query_graph_text`) — one engine, identical ranking, no third-party reimplementation. - **Pi PR-boundary review gains a durable missed-event recovery path** (new [REQ-AGENT-058](agents.md#req-agent-058-pr-boundary-review-reconciliation-and-missed-event-recovery); [REQ-AGENT-036](agents.md#req-agent-036-pr-boundary-review-trigger-conditions) AC7, [REQ-AGENT-040](agents.md#req-agent-040-pr-boundary-lane-classification-and-agent-dispatch) AC8). - **Pi review-enforcement subsystem hardened against the stale-ctx flood and reviews that silently never fired** ([REQ-AGENT-055](agents.md#req-agent-055-pi-pr-boundary-review-window-advancement), [REQ-AGENT-036](agents.md#req-agent-036-pr-boundary-review-trigger-conditions); new [REQ-AGENT-057](agents.md#req-agent-057-pi-review-status-command)). - **Claude PR-boundary review-reminder hook fired inconsistently on multi-line pushes** ([REQ-AGENT-036](agents.md#req-agent-036-pr-boundary-review-trigger-conditions)). `git-push-review-reminder.sh` only recognised `git push` at start-of-command or after a `;` / `&` / `|` separator; a bare newline was not in the separator class, so a `git push` on its own line inside a multi-line Bash command emitted no in-turn review directive (the turn-end Stop hook still caught it, so reviews were eventually enforced — but the in-turn nudge was lost).
