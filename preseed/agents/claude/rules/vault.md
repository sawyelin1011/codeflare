# Vault (persistent obsidian-style notes)

`/home/user/.obsidian_vault/` is the persistent, user-curatable note
store. It survives sessions via rclone bisync to R2 — anything written
here is available on the next session and on every device.

Pair with [memory.md](./memory.md) for the chat-capture half of the
cross-session memory story; see `documentation/vault.md` for full
mechanics.

## Layout

```
.obsidian_vault/
├── raw/
│   ├── sessions/     <- AGENT-OWNED: written by the capture sonnet,
│   │                    one .md per 15-prompt batch. Append-only.
│   └── pasted/       <- USER-OWNED: drag-drop screenshots, PDFs, etc.
├── notes/            <- USER-OWNED: curated prose, concept notes.
├── graphify-out/     <- DERIVED: regenerated by the extract sonnet.
│                        Treat as build output; never hand-edit.
└── .silverbullet/    <- EDITOR CONFIG: SilverBullet's config + plug cache.
```

## Reading from the vault

Prefer the unified graph (`mcp__graphify__*`) for any cross-session
lookup. The graph ingests every vault file plus every active repo's
graphify-out, so a single query returns matches from the vault and
code side by side.

Use `mcp__graphify__query_graph` for free-text questions and
`mcp__graphify__get_node` when you already know the concept name. Only
read raw vault files when the graph query points you at a specific path
and you need the surrounding prose.

## Writing to the vault

| Path | Who writes | When |
|---|---|---|
| `raw/sessions/*.md` | The capture sonnet only | Triggered by the memory-capture hook every 15 user messages. Never hand-edit; the file is deterministic and re-running the agent regenerates it. |
| `raw/pasted/*` | The user only | Image/PDF/asset drops from the SilverBullet UI. Do not generate files here from agent code. |
| `notes/**/*.md` | The user, primarily | If the user asks you to file a note, write here. Use `[[wikilinks]]` for concepts; leave file paths and code symbols as prose. |
| `graphify-out/` | The graphify CLI only | Build output of `graphify extract` / `graphify global add`. Never write directly. |
| `.silverbullet/` | The editor only | SilverBullet maintains it. Touch only if you are explicitly debugging the editor. |

## Wikilink convention

- Wrap **concepts** in `[[wikilinks]]` (e.g. `[[VaultMonitorDaemon]]`,
  `[[GraphifyGlobalAdd]]`, `[[RetryWithBackoff]]`). Graphify's external-
  label dedup unifies these across the vault and per-repo code graphs
  so a single concept node aggregates everywhere it is mentioned.
- Keep **file paths**, **code symbols**, and **issue/PR references** as
  prose — they namespace per-project and auto-linking them would create
  noisy cross-repo collisions.

## Things to NEVER do

- **Never `rm` from the vault.** User-curated content is sacred; if you
  think a file is wrong, ask the user.
- **Never edit `raw/sessions/`.** Agent-owned. Hand-editing creates
  drift between the capture sonnet's view and what is on disk.
- **Never write into `graphify-out/`.** Build output. Run `graphify
  extract` instead.
- **Never `cd` into the vault and run `git init`.** The vault is synced
  via rclone bisync, not git. Adding `.git/` would confuse the
  graphify-active-repo hook and the bisync filters.

## Vault-monitor hook

A second UserPromptSubmit hook (`vault-monitor-hook.sh`) picks up edits
the user makes in the vault directly. It exits 0 silently on the
overwhelming majority of prompts; when it does emit
`additionalContext`, spawn the background sonnet per the directive
(it reads `vault-extract-prompt.md` and runs the 5-step extraction
contract). The sonnet's first step is to delete the marker file, which
is the dedup gate — same protocol as the memory-capture hook.

## Worker proxy boundary

The SilverBullet editor inside the container is reachable from the
codeflare UI through `/api/vault/:sid/`. Auth + rate-limit happens at
the Worker; the SilverBullet server itself is bound to localhost only.
Do not try to expose port 3030 directly — the auth boundary is the
Worker route.
